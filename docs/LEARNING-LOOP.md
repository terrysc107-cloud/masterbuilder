# Learning Loop

This file explains how the builder system becomes smarter across projects and phases.

## Purpose

The team should not only build. It should learn.

That means every completed phase should improve:
- prompts
- templates
- acceptance standards
- reusable code patterns
- execution order judgment

## The Learning Steward Cycle

After each phase, the Learning Steward records:
- what went smoothly
- what was unexpectedly hard
- what mistakes repeated
- what decisions should be remembered
- what prompts need refinement
- what templates should change

## What To Capture

### Reusable Wins

- good component patterns
- effective prompt structures
- strong review checklists
- useful repo organization ideas

### Repeated Problems

- unclear handoffs
- blurry phase boundaries
- overbuilt solutions
- missing design reviews
- weak verification

### Decision Memory

Track decisions such as:
- why a backend was delayed
- why a content model stayed file-based
- why a premium path was structured a certain way
- what tradeoffs were accepted

## Suggested Files To Maintain In Real Product Repos

- `memory/DECISIONS.md`
- `memory/LEARNINGS.md`
- `memory/DEBT.md`
- `memory/PATTERNS.md`

## Improvement Rule

Do not just record what happened. Update the system where possible:
- tighten prompts
- sharpen templates
- improve phase playbooks
- improve the review bar

That is how one build teaches the next build.
