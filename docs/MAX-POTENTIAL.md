# Max Potential

This file explains how to use the builder system at its highest level.

## What "Max Potential" Means

Using this system well means:
- the right intelligence shows up at the right time
- the product gets better in the correct order
- the system learns from each phase
- future builds improve because past builds were documented honestly

## The Biggest Mistake To Avoid

Do not use this system as a fancy brainstorming kit.

Its best use is as an execution governor:
- it decides what matters now
- it prevents premature work
- it forces truth before motion
- it preserves coherence while the product grows

## How To Get Full Value

### 1. Treat the Chief Builder as a real gatekeeper

The Chief Builder should:
- choose the active phase
- freeze unrelated work
- reject clever but mistimed ideas

### 2. Always let the Scout go first

If you skip the audit, the rest of the system becomes guesswork.

### 3. Keep phases emotionally boring

That is a good sign.

When the system is working, each phase should feel focused, bounded, and unsurprising.

### 4. Use the Learning Steward seriously

After each phase:
- record mistakes
- record patterns worth reusing
- improve prompts
- improve templates

### 5. Protect build order

Respect the sequence:
- clarity before polish
- one strong core path before broad expansion
- monetization after product truth
- backend after flow stability
- hardening before launch

### 6. Make decisions visible

Use memory files aggressively:
- `memory/DECISIONS.md`
- `memory/LEARNINGS.md`
- `memory/DEBT.md`
- `memory/PATTERNS.md`

### 7. Reuse the system across projects

This repo gets more valuable when:
- you copy it into new product repos
- you adapt it
- you improve it
- you bring the improvements back here

## Best Working Rhythm

1. Audit
2. Contract
3. Build
4. Review
5. Sign off
6. Learn
7. Advance

## When To Add More Specialists

Add specialist roles only when they are truly needed.

Good examples:
- SEO strategist
- motion designer
- ML systems planner
- growth systems strategist

Do not add specialists just because it feels sophisticated.

When in doubt:
- start without the specialist
- let the Scout identify the real constraint
- let the Chief Builder decide whether a specialist is justified

## Strong Prompt Pattern For High-Leverage Use

```text
Execute Phase 04 using phases/PHASE-04-core-experience-completion.md and follow docs/MASTER-PLAN.md.
Act as Chief Builder, Scout, Frontend Builder, Flow Architect, and Verifier.
Audit first, then implement only what is required for the active phase.
At the end, produce a review verdict and a learning update.
```

## What The System Becomes Over Time

At full strength, this repo becomes:
- a reusable builder brain
- a project execution operating system
- a memory-preserving design and engineering discipline
- a way for multiple LLMs to collaborate without dissolving into noise
